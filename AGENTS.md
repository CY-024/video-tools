## 项目大概是做什么的

这是一个面向自媒体视频创作的网页工具原型，主题是“爆款解说视频工作台”。

它看起来像一个 AI 视频创作后台，主要围绕这些事情：

- 学习热门视频文案结构。
- 根据创作需求生成新的解说文案。
- 把文案和画面片段做匹配。
- 模拟剪映一样的剪辑工作台，包括媒体库、播放器、字幕、音频和时间线。
- 尝试从网页、B 站、抖音链接中读取公开文字或字幕，再做文案学习。

目前它不是完整的真实剪辑软件。更准确地说，它是“可交互的前端展示原型 + 一个轻量 Node 后端”。页面能点、能切换、能展示学习和生成结果；后端能保存学习记录，也能在配置火山方舟模型后调用 AI。

## 技术栈

- React：用来搭网页界面的工具。
- Vite：让网页打包、预览的工具。
- lucide-react：一套现成的小图标库。
- Express：Node.js 后端服务，用来提供接口和托管打包后的网页。
- dotenv：读取 `.env` 里的本地配置，例如 API Key。
- 火山方舟 Ark：可选的 AI 模型接口，没有配置时会走本地兜底规则。

## 主要文件结构

- `package.json`：项目配置，里面写了启动、打包、预览命令和依赖。
- `package-lock.json`：依赖锁定文件，记录安装包具体版本，通常不要手动改。
- `index.html`：网页入口，负责挂载 React 页面。
- `src/main.jsx`：React 启动入口，把 `App.jsx` 渲染到网页里。
- `src/App.jsx`：主要页面逻辑和界面内容都在这里。
- `src/App.css`：页面样式，控制颜色、布局、按钮、面板、剪辑时间线等视觉效果。
- `server/index.js`：Express 后端入口，提供 API，也会托管 `dist/` 打包成品。
- `server/learningService.js`：文案学习和文案生成的核心逻辑；有 Ark 配置时调用模型，没有配置时用本地规则兜底。
- `server/arkClient.js`：火山方舟接口调用封装。
- `server/storage.js`：学习记录读写逻辑，把数据存到 `data/learn-library.json`。
- `server/webReader.js`：读取普通网页的标题、描述和正文文字。
- `server/videoReader.js`：尝试读取 B 站、抖音视频的公开字幕或口播文本。
- `data/learn-library.json`：学习库数据文件，初始可以是空数组 `[]`。
- `dist/`：打包后的网页成品，由 `npm run build` 生成，一般不要手动改。
- `node_modules/`：第三方依赖目录，不要手动改。
- `.env.example`：环境变量示例，不放真实密钥。
- `.env`：本地私密配置，已经被 `.gitignore` 忽略，不要随便打开、复制或提交。
- `.gitignore`：告诉 Git 哪些文件不用记录。

## 当前运行方式

在项目根目录运行：

```bash
npm run dev
```

当前脚本会先执行 `npm run build`，再执行 `npm run server`。也就是说，它会先把前端打包到 `dist/`，再用后端服务打开成品页面。

```bash
npm run build
```

只打包前端页面，生成 `dist/`。

```bash
npm run server
```

只启动后端服务。默认地址是：

```text
http://127.0.0.1:8787
```

```bash
npm run preview
```

当前等同于 `npm run server`，也是通过后端预览打包后的页面。

还有两个脚本：

- `npm run dev:api`：只启动后端接口。
- `npm run dev:web`：用 Vite 预览前端，地址固定为 `http://127.0.0.1:5173`。

## 环境变量

`.env.example` 里列出这些配置：

- `ARK_API_KEY`：火山方舟 API Key。
- `ARK_MODEL_ID`：火山方舟模型 ID。
- `ARK_BASE_URL`：火山方舟接口地址，默认类似 `https://ark.cn-beijing.volces.com/api/v3/chat/completions`。
- `API_PORT`：后端端口，默认 `8787`。

注意：不要把真实 API Key 写进文档、提交到 Git，或者发到聊天里。

## 后端接口

`server/index.js` 当前提供这些接口：

- `GET /api/health`：健康检查。
- `GET /api/learn/records`：读取学习库全部记录。
- `GET /api/learn/samples`：读取最多 8 条学习样本。
- `POST /api/learn/analyze-text`：分析手动粘贴的文案并保存。
- `POST /api/learn/analyze-url`：读取普通网页公开文字，分析后保存。
- `POST /api/learn/analyze-video-url`：尝试读取 B 站或抖音视频公开字幕，分析后保存。
- `POST /api/write/generate`：基于学习库生成新解说文案和 SRT 草稿。

后端也会托管 `dist/`。所以访问 `http://127.0.0.1:8787` 时，看到的是打包后的前端页面。

## 数据保存逻辑

- 学习记录保存在 `data/learn-library.json`。
- 如果文件不存在，`server/storage.js` 会自动创建，并写入 `[]`。
- 每次保存新记录，会把新记录放到最前面。
- 学习库最多保留 200 条记录。

## AI 和兜底规则

如果 `.env` 里配置了 `ARK_API_KEY` 和 `ARK_MODEL_ID`：

- 学习文案会调用火山方舟模型分析。
- 生成文案会调用火山方舟模型生成。
- 模型被要求返回 JSON，代码会解析 JSON 后展示到页面。

如果没有配置：

- 项目不会直接崩掉。
- `learningService.js` 会用本地规则做简单分析和生成。
- 这适合演示流程，但生成质量不能当成真正 AI 效果。

给 Rowan 解释时可以说：没有接上真正的 AI 时，它会用“简易模板”先把流程跑起来，像拍视频前先用占位素材试剪。

## 页面功能模块

`src/App.jsx` 里有 4 个主要模块：

1. `learn`：爆款文案学习
   - 可切换“视频解说提取”“粘贴文案”“网页链接分析”。
   - 能把学习结果保存到学习库。
   - 会展示钩子、冲突、反转、互动等结构拆解。

2. `write`：文案生成
   - 输入创作需求。
   - 参考学习库生成新文案。
   - 展示分段文案和 SRT 字幕草稿。

3. `match`：智能匹配剪辑
   - 展示原视频导入区域。
   - 展示 B-roll 推荐。
   - 展示文案和画面一对一匹配表。
   - 底部会出现多轨时间线。

4. `edit`：剪辑工作台
   - 当前更像剪映工作台入口展示。
   - 页面样式里已经有媒体库、播放器、调节面板、时间线等较完整的 CSS。
   - 但当前 `App.jsx` 里的 `EditView` 实际展示内容比较少，很多剪辑工作台样式还没有完全接到 JSX 上。

## 当前已知问题

- `AGENTS.md`、`index.html`、`src/App.jsx`、`server/*.js` 等文件里存在大量中文乱码。
- 乱码大概率是文件编码问题，类似“中文用错打开方式”，不一定代表逻辑本身错了。
- `AGENTS.md` 已重新整理为正常中文，但不要顺手大范围改其他乱码文件，避免影响现有代码。
- `index.html` 的 `<title>` 当前显示为乱码。
- `src/App.jsx` 里的大量中文 UI 文案是乱码。
- `server` 里的报错提示和模型提示词也有乱码。
- `.env.example` 里的中文占位文字也是乱码。
- `data/learn-library.json` 当前是空数组。
- 项目没有自动化测试脚本。
- 页面 `body` 设置了较大的 `min-width`，偏桌面端，不太适合手机小屏幕。
- `package.json`、`server/index.js` 当前已有未提交改动，接手时不要误以为都是自己改的。

## 修改代码时的注意事项

- 优先修改 `src/App.jsx`、`src/App.css`、`server/` 里对应功能文件。
- 不要手动修改 `node_modules/`。
- 不要手动修改 `dist/`，除非明确是在处理打包结果。
- 不要提交 `.env`，也不要把真实 API Key 写到任何说明文件里。
- 修改中文文字时要确认保存为 UTF-8，避免中文再次变乱码。
- 如果只想修文案乱码，建议小范围、分文件修，不要同时重构逻辑。
- 如果改了界面，最好运行 `npm run build` 确认没有打包错误。
- 如果改了后端接口，最好启动 `npm run server` 后访问 `/api/health` 看是否正常。
- 如果要给 Rowan 解释样式变化，要说“页面看起来会怎么变”，比如：
  - “左边菜单会变窄一点。”
  - “按钮颜色会更明显。”
  - “时间线区域会更像剪映。”
